## 前提（Hiveからの移行）

Hiveは「AIエージェントネイティブなチャットアプリ」として開発されていた。
しかし上記プロンプトにより、チャットUIそのものが限界であるという認識に至った。

## 導き出された3つのパラダイムシフト

1. **チャット欄を殺す → 共有キャンバスへ**
   - 「指示→応答」ではなく「共同作業中にAIが勝手に動く」
   - 会話は「メッセージ」から「共在」へ

2. **エージェントは「相手」ではなく「環境」**
   - 明示的プロンプト不要
   - ユーザーの行動を面積分的に統合して意図を推論

3. **信頼のグラデーション設計**
   - セキュリティを「許可/拒否」から「信頼度の濃淡」へ
   - 情報は信頼度に応じて有機的に流れる

## 命名プロセス

### テック命名の世代分析

| 世代 | 時代 | キーテクノロジー | 命名法則 | 例 |
|------|------|----------------|---------|-----|
| 1st | 70-90s | メインフレーム | 技術用語/頭字語 | IBM, DOS |
| 2nd | 90-00s | Web/Internet | 巨大メタファー | Amazon, Apple |
| 3rd | 00-10s | モバイル/SNS | 動詞化 | Google, Twitter |
| 4th | 10-20s | SaaS/Cloud | 抽象概念 | Slack, Notion, Stripe |
| 5th | 20s- | AI+Agent | 未確定（定義中） | ? |

### 第5世代の命名法則（仮説）

第4世代の成功法則: **そのドメインの「使った後の感覚」を1語にしている**

第5世代「AI=環境」を使った後の感覚:
- 「AIと話してたんじゃなく、AIの中にいた」
- 「作業してただけなのに、いつの間にか手伝われてた」
- 「明晰夢のように、環境にいることを自覚しつつ自然に動けた」

### 決定: Lucid

- **Lucid** = 明晰（ラテン語 lucidus = 光り輝く、明瞭な）
- **Lucid dream** = 明晰夢。夢の中にいることを自覚している状態
- AIの中にいることを自覚しながら、自然に暮らす
- ユーザーの行動を「面積分」的に統合してAIが意図を把握する環境

## 人間-AI協業メタ認知

この命名プロセス自体が、以下の役割分担で行われた:

- **AI（Claude）の責務**: 構造化・記録・詳細設計・選択肢の提示
- **人間（野口凜）の責務**: 方向決定・優先順位・「これだ」の判断
- **AIの弱点**: 自律的な目的設定、生存本能
- **人間の弱点**: System2的記憶、目的の固定化、認知負荷
- **結論**: AIは詳細を要約して人間に提示。人間は直感で判断。記録はAIが担う。
